## C1 — Derived work budget `C` (the default `df_budget`)

**Why.** The Cantelli stop already targets `ln(N/k)` nats (evidence-space). `df_budget` is the same
identification requirement in work-space (postings scanned). Setting it independently double-specifies
the problem; deriving it from the stop closes the loop (the Lagrangian dual of §5's knapsack). See the
exchange-rate derivation: scanning `df` postings buys `≈ ln(N/df)` nats, so the work to collect the
target is

```
C  =  (1/r) · ln(N/k) · d̄ / ln(N/d̄)
       └reliab┘ └target┘ └cost-per-nat of available grams┘
```

with `r = σ`, `k = k_target`, `d̄` = a representative surviving-gram df.

**Where.** `search.rs::prepare` (around the `SelectParams` build at `search.rs:563`), which already
has `n_segments` (N), `sigma`, `k_target`, and the query's `ClassSnap`. Today it passes
`df_budget: opts.df_budget` (`search.rs:567`) straight through. New:

```rust
let derived_c = derived_budget(n_segments, opts.k_target, sel_sigma, &class_snap);
let df_budget = opts.df_budget.or(derived_c);   // user override wins; else derived; else None
```

**`d̄` from corpus stats.** `welford.rs` already maintains per-`(script,order)` `Welford` over `ln df`
(`mean = E[ln df]`). Add a pooled accessor on `ClassSnap`:

```rust
// pool the query's present classes by sample count n: returns (mean_log_df, std_log_df)
pub(crate) fn pooled_log_df(&self) -> Option<(f64, f64)>;
```

Then `d̄ = exp(mean_log_df + Z · std_log_df)`, with **`Z ≥ 0` a shape constant** (Z = 0 → geometric
mean; Z > 0 → a commoner percentile → larger, safer `C`). See **Open decision A** — `Z` is the one
constant M6 must pin, and it is a *recall-safety* knob, not a perf knob.

**Guards (recall-safe fallbacks — when unsure, go unbounded):**
- pooled stats unavailable (every present class has `n < 2`, `std = ∞`) → `C = None` (unbounded).
- `N ≤ k_target` (tiny corpus, no pruning meaningful) → `C = None`.
- clamp `d̄ ∈ [1, N−1]`; require `ln(N/d̄) > 0` and `ln(N/k) > 0`, else `None`.
- result clamped to `u64`; `C = None` if it overflows / is non-finite.

**⚠ Recall-safety constraint (the heart of the spec).** The per-class floor and typo floor `F` admit
only `df ≤ C` grams (`select.rs:244,263`); grams with `df > C` are skipped (`§5`'s bounded recall
cost). So **if `C` is set *below* the typical real query-gram df, real rare grams get skipped → recall
loss.** A geometric-mean `d̄` (Z = 0) can land `C ≈ O(k) ≈ 128`, which would starve any query whose
surviving grams are commoner than df 128 — i.e. most fuzzy queries. Therefore `Z` must be tuned
*upward* so `C` sits at/above the recall knee, and the safe default **errs large** (toward unbounded).
This is exactly what the validation (Open decision B) must confirm; it is why this can't ship on an
un-validated constant.

**New private fns in `search.rs`:** `derived_budget(n, k, r, &ClassSnap) -> Option<u64>` (+ unit tests
on the formula, the guards, and a monotonicity check: larger `d̄` ⇒ larger `C`).

## C2 — Remove `t_max`

`t_max` is the last corpus-tuned magic number in selection and is redundant once C1 makes the budget
load-bearing: count is already bounded by the query's finite gram set, work by derived-`C`. Selection
becomes `F + per-(script,order) floors + rarest-first until (Cantelli stop ⊓ C)`.

- **`src/lib.rs`:** delete `SearchOpts.t_max` field (`lib.rs:197`), its builder, and `DEFAULT_T_MAX`.
- **`src/select.rs`:** delete `SelectParams.t_max`; in `select()` remove the `ceiling` computation
  (`select.rs:232`) and the `n_kept >= ceiling` backstop (`select.rs:281–283`). The greedy loop now
  runs over all present grams, bounded only by `acc.fired(...)` and the budget skip.
- **`src/search.rs::prepare`:** drop `t_max` from the `SelectParams` build (`search.rs:563`).
- **Tests:** update every `select.rs` test using the `params(min_shared, t_max)` helper
  (`select.rs:328`) and any `SearchOpts::t_max` usage. The helper loses its `t_max` arg.

**Breaking API change** (pre-1.0; `SearchOpts` is `#[non_exhaustive]`/builder-built, so callers using
`.t_max(n)` get a compile error and drop the call — clean).

## C3 — Exclude floored grams from the §9 cap's `E_top`

`concentration_cap` (`search.rs:441`) currently folds *all* energies, so a floored junk gram at
`E_max` can be `E_top` and *loosen* the cap, defeating its purpose (protect the real discriminating
gram). Change it to key off **non-floored** grams only.

- **Signature:** `concentration_cap(energies: &[f64], floored: &[bool]) -> Option<f64>` (or pass a
  pre-filtered slice). Filter floored out of the `e_top`/`commons` computation.
- **Call site `search.rs:958`:** pass the parallel `present_floored` mask (already available on the
  plan, `search.rs:184`).
- **Tests:** flip `concentration_cap(&[6.9, 0.0, 0.0])` "floored gram is a valid dominant"
  (`search.rs:1915`) to assert the floored top is **excluded**; add the R1 regression (floored `E_top`
  no longer loosens the cap → the on-topic rare-gram doc is not out-ranked).
- **Doc:** update the `concentration_cap` interpretation note (`search.rs:421–440`) from "literal §12,
  floored included" to "floored excluded," and amend `docs/derivation.md` §9/§12 accordingly.

## C4 — `Candidate` energy / count / length / nat_score accessors (§10) — for downstream fusion

**Motivation (from the downstream consumer, `../ruffle`).** ruffle fuses trifle (a lexical channel)
with others via calibration-free weighted RRF. It takes **one scalar per item** and derives both a
rank (`D^sep`) *and* an absolute goodness (`D^abs`) that compares the channel's top scores **against a
reference "in native units"** (`ruffle/src/config.rs:190`), declared or learned-from-traffic. So it
needs a **stable, cross-query-comparable nat-scale magnitude.** trifle's `corrected_score` is **not**
that — it flips scale by query (nat-scale single-view vs RRF-scale `≈0.01–0.03` fused; `search.rs:132–143`
says so outright), which would poison a learned/declared `D^abs` reference.

**Ship all four accessors** (decision C, ratified), documented as usable separately:

- `energy()`, `count()`, `length()` — the §10 components, in **nats**.
- `nat_score() = energy + count − length` — trifle's length-corrected relevance magnitude, the
  **fusion-ready `D^abs` input** (cross-query comparable by construction). For a single-view query it
  *equals* `corrected_score()`; for a fused query it is the stable nat-scale magnitude while
  `corrected_score()` remains the within-query RRF rank key (unchanged — do not touch the ranking).

**Governing view (the "fix" to `energy()`): the best-ranked (retained) view, never a cross-view sum.**
The code already embodies no-summing: `rrf_fuse` (`search.rs:1368`) retains, per key, the **best-ranked
view's candidate clone** (`slot.0 = c.clone()` when `rank < slot.2`), whose `score`/`E_acc` is that one
view's energy (doc `search.rs:1363–1365`; test `rrf_reads_ranks_not_summed_energy_and_omits_the_absent`).
Summing `E_primary + E_secondary` would double-count the same text at two granularities — exactly what
RRF exists to prevent. So the accessors report the **retained view's** components, which also keeps
energy/`count`/`length` coherent with the candidate's **provenance** (`label`/`seg_id`/span, owned by
that same retained clone). Choosing "primary" or "max-energy" instead would let a candidate report
energy from a different view than the span it matched — incoherent, rejected.

- **`Candidate`** (`search.rs:102`) currently holds `score: u32` (`E_acc`) + `corrected_score: f64`.
  Add `energy: f64`, `count: f64`, `length: f64` (all nats), computed at score-time in `score_union`
  (from `ScorePlan::corrected`'s parts, `search.rs:270`: `energy = e_acc·Δ`, `count = credit`,
  `length = null`). The retained fused clone carries its own (best-ranked) view's values — no fusion-time
  accumulation, no sum.
- `nat_score()` recomputes `energy + count − length` from the stored parts.
- **Tests:** single-view `nat_score() == corrected_score() == energy() + count() − length()`; component
  values match a hand-computed fixture; a fused candidate's `energy()` equals its retained view's
  `score()·Δ` (proving no cross-view sum); `nat_score()` is nat-scale for both single-view and fused.

## Open decisions

**A. `Z` (the `d̄` percentile shape constant) — RESOLVED: `Z = 2.0`** (≈ P98 of `ln df`). Balanced
conservative: recall-safe with margin, still caps the common-gram tail so derived-`C` bounds work in
0.4.0. The recall-guard test (B) enforces it upward if it regresses.

**B. Validation path — RESOLVED: conservative `Z` now + in-tree recall-guard as the 0.4.0 gate;
selsweep `Z`-tuning deferred to 0.4.1.** 0.4.0 ships the derived-`C` default-shape (frozen) with
`Z=2` gated by a `tests/` recall-guard (synthetic corpora at a few `N`, asserting derived-`C` recall ≥
unbounded − ε). Reviving `benchmarks/` selsweep to tune `Z` down to the measured knee is a
**recall-preserving latency win → 0.4.1** (perf-only; no API/default-shape change, so it fits the
0.4.1 remit).

**C. `Candidate` components — RESOLVED: ship all four** (`energy`/`count`/`length`/`nat_score`),
governing-view (best-ranked/retained), never summed. See C4 above.

**D. Opportunistic #10(a) — RESOLVED: defer to 0.4.1** (paired with the `Z`-tuning that makes the
budget actually bite), folding into M6 only if it falls out as a trivial few-line change in the
`select.rs` budget path being reworked. Default: **deferred**.
