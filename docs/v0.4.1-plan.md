# v0.4.1 — performance-only follow-up

**Remit: performance only. No API churn, no scoring-behavior change, no on-disk format change.**
0.4.0 froze and finalized the public surface and the scoring behavior (see `docs/derivation.md`, the
authoritative spec, and `docs/v0.4-M6-plan.md`, the 0.4.0 finalization slice). 0.4.1 makes the same
results faster; anything that would move a ranking, change a default's *shape*, or alter a stored blob
belongs to a later minor, not here.

The two invariants that bound every item below: **recall is preserved** (0.4.1 must not drop a
candidate 0.4.0 would have returned) and **flatness holds** (engine op count stays
cardinality-independent). Each item names how it stays inside both.

---

## Headline win — tune the derived work budget `C` down to the recall knee

**What.** 0.4.0 ships the derived default `C = (1/σ)·ln(N/k)·d̄/ln(N/d̄)` with `d̄ = exp(mean_lndf +
Z·std_lndf)` at a **conservative `Z = 2.0` (≈ P98)** — deliberately erring large so `C` never starves a
real query's grams (the recall-safety constraint; see `derived_budget` in `src/search.rs` and §5/§7 of
the derivation). A large `Z` means a larger `C`, so more postings scanned per query than strictly
needed. Tuning `Z` **downward** toward the measured recall knee is a pure **latency win that preserves
recall** — fewer postings for the same candidate set — which is exactly the 0.4.1 remit (the
default's *shape* — the formula, its guards, the `df_budget` override — is unchanged; only the one
shape constant `Z` moves, and only as far as the data proves safe).

**Why it's deferred, not done in 0.4.0.** It can't ship on an un-validated constant. Finding the knee
needs the `benchmarks/` **selsweep** harness (recall@k vs Σdf / p99 latency across an N-ladder), which
is currently **excluded from the workspace** and does not build against the streaming candidate API.

**Work.**
1. **Revive `benchmarks/`** against the current `Reader::candidates` / `CandidateStream` API (it is not
   part of any gate today; getting it building is the prerequisite). See the selsweep methodology memo.
2. Run the selsweep across the N-ladder `{1k, 5k, 25k, 125k, 625k}`, sweeping `Z ∈ [0, 2]`, plotting
   recall@k against Σdf and p99 latency; locate the knee where recall is still ≈ 1.0.
3. Lower the `Z` default to the validated knee. **Gate:** `tests/recall_budget.rs` (the in-tree
   recall-safety gate) must stay green at the new `Z`; extend it with the ladder points the sweep
   exercises so a future `Z` change can't silently regress recall.

**Guard rail.** `Z` is a recall-safety knob, not just a perf dial. The recall-guard test is the hard
gate; the sweep only tells us *how far down* we may push before it trips.

## Post-pass / walk micro-costs (bounded, flatness-safe)

Both are known constant-factor costs the M3/M5 review panels flagged as non-blocking. Neither changes
the candidate set or the scores.

- **Bounded top-`k` in the eager early-stop window.** The eager `matches`/`matches_batch` path can hit
  a worst case of `O(union²/CHUNK)` on deep-`k` queries where the stop does not fire (≈ 0 cost at
  shallow `k`; the cost is cardinality-independent, so flatness already holds, and it is
  SQL-dominated in practice). The `kth_largest` recompute is the culprit (see the deferral note left in
  `src/search.rs`). Replace it with a bounded top-`k` **min-heap** / memoized `kth_best` so the window
  is `O(union·log k)`. Pure post-pass; the returned top-`k` is identical.
- **Fuse the walk's double `contains`.** A mixed-weight *walk* candidate currently pays a per-gram
  `contains` twice — once for the engine's `raw_overlap` `min_shared` floor-gate, once for the count
  credit. The count-only path already fuses these into a single pass (`overlap_and_credit`); extend the
  same fusion to walk candidates. This is the inherent seam between the N-free engine and the
  N-anchored credit post-pass; fusing it removes one `O(k)`-per-candidate scan with no behavior change.

## Space (measure, don't necessarily fix)

Dual-order tokenization (M5) roughly **2.0–2.5×**'s posting/dictionary storage vs a trigram-only
index — by design (§8), the cost of the secondary rank-view. 0.4.1 should **measure** it on a real
corpus and decide whether any of it is recoverable without touching the on-disk format or losing the
secondary view (e.g. tighter blob packing, dictionary interning wins). Anything that changes the stored
format is *not* 0.4.1 (it would force a drift-reset); this bullet is measurement + format-preserving
packing only.

## Explicitly NOT in 0.4.1 (they change behavior or format — later minors)

- **Precision refinements** (`V04_HANDOFF.md` flags #5, #10a/b): delimiter-level word-breaking (finer
  than the recall-safe whitespace-only break); the §12 `loosen_budget_or_restore_reserve` routing that
  restores the mid-rarity energy reserve before dropping to the secondary view; the rescue carve-out's
  reach into the dense unigram view. These move rankings → a precision-focused minor, not the perf-only
  0.4.1.
- **The field-aware index** (per-field posting partitions): unlocks field-scoped candidate-gen,
  field-local idf, the per-field doc-side `ε` channel (`ρ = σ(1−ε)^n`), and first-class field
  filtering. This is an architecture/storage change (a drift-reset), so it is a **0.5**-scale milestone,
  not 0.4.1. Field-scoped *results* already work today via a `SqlFilter` on `seg.label` (see its
  rustdoc). A cheap middle rung — a per-field seg-id bitmap `AND`ed into the gram posting at query time
  — gives field-scoped candidate-gen without full partitioning, but even that touches storage, so it is
  out of the perf-only remit.
